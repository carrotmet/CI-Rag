# 协调器 V1 vs V2 对比摘要

## 核心差异一览

| 维度 | V1 (已有) | V2 (优化后) |
|:-----|:----------|:------------|
| **决策主体** | 协调器自决策 | Level 2 决策，协调器执行 |
| **BD 区要求** | 可无 guide 进入 | **必须有 orchestrator_guide** |
| **missing_info** | 协调器启发式识别 | **Level 2 专业判断** |
| **分解决策** | 协调器规则判断 | **Level 2 结构分析，可与补充并存** |
| **L2 调用量** | ~10% (逃逸) | ~40% (逃逸+BD轻量) |
| **延迟** | BD区直接执行 | BD区 +50ms (轻量L2) |

---

## 架构流程对比

### V1 流程
```
Query
  ↓
L0/L1/L2 → Zone B/D
  ↓
协调器自决策
  ├── 自己识别 missing_info
  ├── 自己判断 decomposable
  └── 选择转区策略
  ↓
执行转区
```

### V2 流程
```
Query
  ↓
L0/L1/L2 → Zone B/D + **强制轻量 L2**
  ↓
生成 orchestrator_guide
  ├── Level 2 识别 missing_info
  ├── Level 2 判断 decomposable
  └── Level 2 推荐策略
  ↓
协调器按 guide 执行
  ├── 不自己决策
  └── 直接执行推荐策略
```

## V2 关键特性：多策略并存

### clarify 与 decompose 可同时存在

**V1 限制**：协调器只能二选一
- `recommended: "clarify"` → 只能补充信息
- `recommended: "decompose"` → 只能分解问题

**V2 改进**：Level 2 提供策略列表
```json
{
  "recommended": ["clarify", "decompose"]
}
```

协调器可以：
1. **优先尝试 clarify** → 如果用户不愿补充信息 → 提供 decompose 选项
2. **同时展示两种方案** → 让用户自主选择
3. **根据上下文动态切换** → 第一轮 clarify 失败，第二轮尝试 decompose

### 策略组合示例

| 策略列表 | 适用场景 | 执行逻辑 |
|:---------|:---------|:---------|
| `["clarify", "decompose"]` | 既可补充又可分解 | 先尝试补充，不成则分解 |
| `["decompose", "clarify"]` | 结构清晰的多部分问题 | 优先分解，子问题再补充 |
| `["clarify"]` | 单一信息缺口 | 仅信息补充 |
| `["decompose"]` | 复杂系统设计 | 仅问题分解 |

## 决策建议

### 选择 V1 如果：
- 快速原型验证
- 对转区策略准确性要求不高
- 资源受限（无法接受额外 L2 调用成本）

### 选择 V2 如果：
- 生产环境部署
- 需要可靠的转区策略
- 可接受 ~50ms 额外延迟换取准确性
- 有缓存机制降低长期成本

---

## 推荐实施路径

1. **短期**：V1 已可用，继续迭代功能
2. **中期**：并行开发 V2 (10-13工作日)
3. **长期**：V2 替代 V1，V1 废弃或保留为轻量模式

---

**结论**: V2 架构更清晰、策略更可靠，值得投入开发。
