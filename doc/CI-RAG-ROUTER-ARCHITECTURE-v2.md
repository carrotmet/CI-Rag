## 详细流程

### 第一阶段：Level 012 逃逸与分区

```
用户查询
    ↓
Level 0 (XGBoost) < 1ms
    ├── σ≥0.7 → 直接输出 Zone + CI
    └── σ<0.7 → 升级
                ↓
        Level 1 (混合检索) ~50ms
            ├── σ≥0.7 → 直接输出 Zone + CI
            └── σ<0.7 → 升级
                        ↓
                Level 2 (裁决)
                    ├── 完整调用: CI仲裁 → Zone + CI + Guide
                    └── 轻量调用: 生成Guide → 返回上级
```

**Level 2 两种模式选择**：
- **完整调用** (~100ms): 用于 CI 置信度低，需要深度仲裁的场景
- **轻量调用** (~50ms): 仅生成 Guide，不重新判定 CI

---

### 第二阶段：协调器介入

#### 功能1：Guide 检查与生成

```
Level 012 输出
    ↓
协调器接收
    ↓
Zone = A/B/D ?
    ├── 是 → 检查 Guide 存在性
    │           ├── 有Guide → 直接放行到对应Zone处理
    │           └── 无Guide → 调用轻量L2生成Guide → 再放行
    └── 否(Zone C) → 直接放行到C区
```

#### 功能2：转区路由与校验

**Zone C 处理**:
```
问题进入Zone C
    ↓
直接推理输出
```

**Zone D 处理**:
```
问题进入Zone D (C=0, I=0) + Guide
    ↓
执行检索 (根据Guide.missing_info)
    ↓
请求协调器转区
    ↓
协调器校验:
    ├── I ≥ 阈值 → 转 Zone C 输出
    └── I < 阈值 → 打回 Zone D 继续检索
```

**Zone A 处理**:
```
问题进入Zone A (C=1, I=1) + Guide
    ↓
根据Guide拆分子问题 [SP1, SP2, SP3] (预期C0I1)
    ↓
协调器校验每个子问题的C (Level 0快速验证):
    ├── C=1 (不达标) → Zone A内部继续拆解该子问题
    └── C=0 (达标) → 保留在子问题列表
    ↓
子问题列表 + 整合提示 → 请求协调器转区
    ↓
协调器转Zone C (子问题整合模式)
    ↓
Zone C 综合推理整合 → 直接输出
```

**Zone B 处理**:
```
问题进入Zone B (C=1, I=0) + Guide
    ↓
根据Guide策略:
    ├── 能检索 → 先执行检索 (提升I)
    └── 无法检索 → 拆解问题 (降低C)
    ↓
迭代执行直到达标
    ↓
C降低/I增加达标后 → 请求协调器转Zone C
    ↓
Zone C接收: 子问题列表 + 辅助信息 + "B区组装"标签
    ↓
综合推理 → 触发"组装整合" → 输出
```

---

### 第三阶段：Zone C 多模式输出

Zone C 作为唯一出口，支持三种模式：

#### 模式1：普通直接输出
```
适用: 原始查询就是Zone C (C0I1)
处理: 直接推理 → 输出
```

#### 模式2：子问题整合输出 (来自Zone A)
```
输入: 子问题列表 + 整合提示 (来自Zone A拆解)
处理:
  1. Zone C 综合所有子问题信息
  2. 根据整合提示进行推理
  3. 生成完整答案
输出: 直接输出整合后的完整答案
```

**流程修正**:
```
Zone A 拆解 → 子问题列表 + 整合提示
                ↓
         协调器转区 → Zone C (批量接收子问题)
                ↓
         Zone C 综合推理整合 → 直接输出
```

#### 模式3：组装整合输出 (来自Zone B)
```
输入: 子问题列表 + 辅助信息 (带"B区组装"标签)
处理:
  1. 综合分析子问题和辅助信息
  2. 推理生成完整答案
  3. 触发组装整合
输出: 完整综合性答案
```

## 协调器核心职责详解

### 1. 子问题生命周期管理

```python
class Orchestrator:
    def handle_zone_a(self, query, guide):
        """Zone A 处理: 拆解 → 子问题校验 → 批量转Zone C整合输出"""
        # 1. 拆解
        sub_problems = decompose(query, guide)
        
        # 2. 校验子问题C值，不达标的继续拆解
        valid_subproblems = []
        for sp in sub_problems:
            while True:
                ci = level0_quick_check(sp.query)
                if ci.C == 0:  # 达标
                    valid_subproblems.append(sp)
                    break
                else:  # 不达标，继续拆解
                    sp = further_decompose(sp)
        
        # 3. 批量转Zone C，由Zone C整合输出
        return zone_c.process_subproblems(valid_subproblems, guide.aggregation_prompt)
    
    def handle_zone_b(self, query, guide):
        """Zone B 处理: 检索/拆解迭代 → 达标转C"""
        context = {"retrieved": [], "subproblems": []}
        
        # 迭代优化
        while not is_ready_for_c(query, context):
            if can_retrieve(query, guide):
                info = retrieve(query, guide)
                context["retrieved"].append(info)
            else:
                sp = decompose_for_b(query, guide)
                context["subproblems"].extend(sp)
        
        # 达标后转Zone C (带组装标记)
        return zone_c.process_assembled(query, context)
```

### 2. 校验逻辑

| 校验点 | 校验内容 | 通过标准 | 失败处理 |
|:-------|:---------|:---------|:---------|
| **Zone D 转区** | 检索后的 I 值 | I ≥ 0.7 | 打回D区继续检索 |
| **Zone A 子问题** | 子问题的 C 值 | C = 0 | A区内部继续拆解 |
| **Zone B 达标** | C 降低或 I 增加 | C≤0.3 或 I≥0.7 | 继续迭代检索/拆解 |

### 3. 整合策略

| 来源 | 整合方式 | 说明 |
|:-----|:---------|:-----|
| Zone A 子问题 | 列表格式化拼接 | 保持各子问题独立性 |
| Zone B 子问题 | 综合推理整合 | 需要整体连贯性的答案 |

## 关键设计决策

### 1. 为什么协调器负责子问题校验而非直接走完整Level 012？

| 方案 | 延迟 | 成本 | 可靠性 | 结论 |
|:-----|:-----|:-----|:-------|:-----|
| 完整Level 012 | ~50-150ms | 高 | 最高 | 过度验证 |
| **Level 0快速校验** | ~1ms | 低 | 高 | **推荐** |
| 完全信任 | 0ms | 最低 | 中 | 风险较高 |

**选择Level 0快速校验**：
- 子问题是拆解产生的，已有预期CI
- Level 0仅需验证C=0，成本低
- 不达标记为异常，返回重新拆解

### 2. Zone C 三种输出模式

Zone C 作为唯一出口，通过不同接口方法支持三种场景：

| 方法 | 场景 | 输入 | 输出 |
|:-----|:-----|:-----|:-----|
| `process()` | 原始Zone C | 单查询 | 直接推理结果 |
| `process_subproblems()` | Zone A 子问题整合 | 子问题列表 + 整合提示 | 整合后的完整答案 |
| `process_assembled()` | Zone B 组装 | 原始查询 + 子问题 + 辅助信息 | 综合推理结果 |

**设计原则**: Zone C 是出口，接收所有必要信息后一次性输出，不返回给协调器二次处理。

### 3. 为什么Zone B先检索后拆解？

**策略逻辑**:
- B区特点是 C=1(复杂) + I=0(信息不足)
- 如果能通过检索提升I到Zone A (C1I1)，再拆解更合理
- 避免盲目拆解导致子问题仍信息不足
- 节省子问题数量，降低总体成本
