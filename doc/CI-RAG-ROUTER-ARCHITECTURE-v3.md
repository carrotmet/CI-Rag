## 关键设计变更详解

### 1. Guide 作为公共资源

```
轻量 L2 Guide 生成器 (公共服务)
         │
    ┌────┼────┬────────┐
    │    │    │        │
    ▼    ▼    ▼        ▼
 Zone D Zone A Zone B  子问题
```

**各区域内部调用场景**:
- **Zone D**: 检索后仍不达标，生成新的检索 Guide
- **Zone A**: 子问题拆解后不达标，生成子问题的拆解 Guide
- **Zone B**: 迭代过程中生成新的检索/拆解 Guide
- **子问题**: 嵌套拆解时自动生成轻量 Guide

### 2. 协调器职责简化

| 版本 | 协调器职责 | 复杂度 |
|:-----|:-----------|:-------|
| V1 | Guide生成、CI判定、转区、子问题管理 | 高 |
| V2 | Guide检查、CI判定、转区、子问题整合 | 中 |
| **V3** | **仅: CI检测、转区路由** | **低** |

**V3 协调器伪代码**:
```python
class Orchestrator:
    def route_if_valid(self, query, current_ci, target_zone):
        """
        职责1: 检测转区标准
        职责2: 执行转区
        """
        if self.check_transition_criteria(current_ci, target_zone):
            return self.route_to_zone(query, target_zone)
        else:
            raise ValidationError("CI不达标，继续当前区域处理")
    
    def check_transition_criteria(self, ci, target_zone) -> bool:
        """检测转区标准"""
        if target_zone == Zone.C:
            return ci.C == 0 and ci.I >= 0.7
        return False
```

### 3. 区域内部自治流程

#### Zone A 自治流程

```
原始问题 (C1I1) + Guide
    ↓
拆解 → 子问题列表 [SP1, SP2, SP3]
    ↓
对每个子问题内部校验C (Level 0快速):
    ├── C=1 (不达标)
    │       ↓
    │   内部生成轻量Guide (调用轻量L2)
    │       ↓
    │   继续拆解该子问题
    │       ↓
    │   循环直到达标
    │
    └── C=0 (达标) → 保留
    ↓
全部达标 → 批量请求协调器转C → Zone C整合输出
```

#### Zone B 自治流程

```
原始问题 (C1I0) + Guide
    ↓
迭代处理循环:
    ├── 尝试检索提升I
    │       ↓
    │   内部校验: 达标? → 转C
    │              不达标? → 继续
    │
    ├── 尝试拆解降低C
    │       ↓
    │   内部生成轻量Guide
    │       ↓
    │   拆解子问题
    │       ↓
    │   内部校验: 达标? → 转C
    │              不达标? → 继续
    ↓
达标 → 请求协调器转C
```

## 利弊分析

### 优点

| 方面 | 说明 |
|:-----|:-----|
| **职责清晰** | 协调器只做"交警"（看标准、指方向），各区域做"司机"（具体怎么开） |
| **减少耦合** | 协调器与区域内部逻辑解耦，只通过转区接口交互 |
| **灵活性高** | 各区域可自主决定迭代策略，不需要协调器介入细节 |
| **可扩展性** | 新增区域类型只需实现内部逻辑+达标后调用协调器转区 |
| **简化调试** | 问题定位更清晰：CI不达标是区域内部问题，转区失败是协调器问题 |

### 缺点

| 方面 | 说明 | 缓解措施 |
|:-----|:-----|:---------|
| **L2调用增加** | 各区域内部可能多次调用轻量L2生成Guide | 轻量调用仅~50ms，成本可控 |
| **重复逻辑** | A/B/D区都需要内部Guide生成逻辑 | 抽象为公共组件供各区域调用 |
| **状态分散** | 子问题状态在各区域内部管理 | 区域内部维护清晰的状态机 |

### 权衡结论

**利大于弊**。V3架构符合"高内聚、低耦合"原则：
- 协调器聚焦核心职责（转区路由）
- 各区域自治内部流程（拆解、检索、Guide生成）
- 清晰的边界：达标前区域内部处理，达标后协调器转区

class Orchestrator:

    职责: 转区检测与路由

### Zone A (拆解区)

```python
class ZoneA:
    """
    职责: 拆解问题直到子问题达标
    内部可调用: GuideGenerator
    """
    
    def process(self, query, guide):
        # 1. 根据Guide拆解
        subproblems = decompose(query, guide)
        
        # 2. 内部迭代直到全部达标
        while not all_valid(subproblems):
            for sp in subproblems:
                if not valid_c(sp):
                    # 内部生成Guide继续拆解
                    sub_guide = guide_generator.generate_for_zone_a(sp.query, context)
                    sp = further_decompose(sp, sub_guide)
        
        # 3. 达标后请求协调器转C
        return orchestrator.route_if_valid(query, ci, Zone.C)
```

## 调用链路对比

### V2: 协调器深度参与

```
Zone A 拆解
    ↓
协调器校验C
    ↓
不达标 → 打回A
    ↓
A再拆解
    ↓
协调器校验C
    ↓
...循环
    ↓
达标 → 协调器收集结果 → 整合 → 转C
```

### V3: 区域自治，协调器仅做转区

```
Zone A 拆解
    ↓
A内部校验C
    ↓
不达标 → A内部生成Guide → A继续拆解
    ↓
...循环(完全在A内部)
    ↓
全部达标
    ↓
请求协调器转C
    ↓
协调器检测标准 → 通过 → 转C
    ↓
Zone C 输出
```

## 接口定义 V3

### 协调器接口

```python
@dataclass
class RouteRequest:
    query: str
    current_ci: CIState
    target_zone: Zone
    context: Dict  # 可选上下文

@dataclass  
class RouteResponse:
    success: bool
    zone: Zone
    result: Optional[Response] = None
    error: Optional[str] = None

class Orchestrator:
    def route_if_valid(self, request: RouteRequest) -> RouteResponse:
        """
        协调器唯一职责: 检测转区标准并执行转区
        
        标准:
        - Zone C: C=0 and I>=0.7
        - 其他Zone: 根据具体策略
        """
        pass
```

### 区域公共接口

```python
class ZoneHandler(ABC):
    """各区域处理基类"""
    
    @abstractmethod
    def process(self, query: str, guide: Guide) -> Response:
        """
        区域处理流程:
        1. 根据Guide执行区域逻辑(拆解/检索)
        2. 内部校验CI
        3. 不达标则内部迭代(可生成新Guide)
        4. 达标后调用 orchestrator.route_if_valid()
        """
        pass
    
    @abstractmethod
    def validate_internally(self, query: str, expected_ci: CI) -> bool:
        """内部校验CI是否达标"""
        pass
```
