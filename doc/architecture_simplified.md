# CI-RAG-Router 简化架构文档

## 架构概述

本文档描述 CI-RAG-Router 的简化架构，核心原则：**只有 Zone C 是出口**。

```
┌─────────────────────────────────────────────────────────────────┐
│                      CI-RAG-Router 简化架构                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   查询输入                                                       │
│       │                                                         │
│       ▼                                                         │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                    逃逸层 (Level 0/1/2)                  │  │
│   │                                                         │  │
│   │   ┌─────────┐    ┌─────────┐    ┌─────────┐            │  │
│   │   │ Level 0 │───→│ Level 1 │───→│ Level 2 │ (裁决)      │  │
│   │   │XGBoost  │    │混合检索  │    │ LLM仲裁 │            │  │
│   │   └─────────┘    └─────────┘    └─────────┘            │  │
│   │        σ<0.7          σ<0.7          σ<0.7              │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│       │                                                         │
│       ▼                                                         │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │              处理层 (ABCD 区域) - 仅 Zone C 出口          │  │
│   │                                                         │  │
│   │   ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐   │  │
│   │   │ Zone A  │  │ Zone B  │  │ Zone C  │  │ Zone D  │   │  │
│   │   │ (C1I1)  │  │ (C1I0)  │  │ (C0I1)  │  │ (C0I0)  │   │  │
│   │   │ 拆解区  │  │ 复杂区  │  │  ✅出口  │  │ 补I区   │   │  │
│   │   └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘   │  │
│   │        └────────────┴────────────┴────────────┘         │  │
│   │                         │                               │  │
│   └─────────────────────────┼───────────────────────────────┘  │
│                             │                                   │
│                             ▼                                   │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                        协调器                             │  │
│   │   职责:                                                   │  │
│   │   - ABCD 区域转区决策                                     │  │
│   │   - 检索执行                                              │  │
│   │   - 子问题拆解                                            │  │
│   │   - 大模型交互                                            │  │

│                             │                                   │
│                             ▼                                   │
│   ┌─────────────────────────────────────────────────────────┐  │

│   │                    C=0, I=1 → LLM直接输出                  │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

---

## 逃逸层 (Level 0/1/2)

### Level 0: XGBoost 快速分类
- **目标**: < 1ms 延迟
- **输出**: C (复杂度), I (信息充分性), σ_joint (联合置信度)
- **逃逸条件**: σ_joint < 0.7 → 升级到 Level 1

### Level 1: 混合检索验证
- **目标**: ~50ms 延迟
- **功能**: 向量检索 + 关键词检索 + 结构化检索
- **输出**: 验证并精化 I 值
- **逃逸条件**: σ_joint < 0.7 → 升级到 Level 2

### Level 2: LLM 仲裁 (裁决层)
- **目标**: ~100ms 延迟
- **职责**: 最终 CI 判定
- **输出**: 确定 Zone (A/B/C/D)
- **两种调用模式**:
  - **完整调用**: 深度 CI 评估，返回 C/I
  - **轻量调用**: 生成转区策略，不返回 C/I

---

## 处理层 (ABCD 区域)

| 区域 | C | I | 名称 | 处理方式 |
|:----:|:-:|:-:|:-----|:---------|
| **A** | 1 | 1 | 拆解区 | 问题拆解 → 子问题 → Zone C |
| **B** | 1 | 0 | 复杂区 | 拆解+补I → Zone C |
| **C** | 0 | 1 | **出口** | **LLM 直接输出** |
| **D** | 0 | 0 | 补I区 | 检索补信息 → Zone C |

**核心原则**: A/B/D 三个区域最终都必须转换为 Zone C 才能输出。

---

## 协调器职责

协调器负责处理所有非 Zone C 区域的转区和转换：

### 1. Zone A (C1I1) 处理
```
Query (C1I1)
    ↓
问题拆解
    ↓
子问题1 (C0I1) ──→ Zone C ──┐
子问题2 (C0I1) ──→ Zone C ──┼──→ 聚合结果
子问题3 (C0I1) ──→ Zone C ──┘
```

### 2. Zone D (C0I0) 处理
```
Query (C0I0)
    ↓
检索/用户补充信息
    ↓
C0I1 → Zone C → 输出
```

### 3. Zone B (C1I0) 处理
```
Query (C1I0)
    ↓
协调器决策:
  ├── 先拆解 → 子问题补I → Zone C
  └── 先补I → Zone A → 拆解 → Zone C
```

## 数据流

```
用户查询
    ↓
Level 0 (XGBoost)
    ↓
Level 1 (检索验证)
    ↓
Level 2 (裁决 Zone)
    ↓
┌─────────────────────────────────────┐
│ Zone C? ──→ 直接输出                │
│ Zone A? ──→ 协调器拆解 ──→ Zone C   │
│ Zone D? ──→ 协调器补I ──→ Zone C    │
│ Zone B? ──→ 协调器拆解+补I ──→ Zone C│
└─────────────────────────────────────┘
    ↓
统一出口 (Zone C)
```

## 关键设计决策

### 1. 为什么只有 Zone C 是出口？
- **简化性**: 统一的输出口简化系统设计
- **一致性**: 所有查询经过相同的最终处理逻辑
- **可维护性**: 只需维护一种输出模式

### 2. 协调器的角色
- **不决策 CI**: CI 判定由 Level 0/1/2 完成
- **执行转区**: 根据 Level 2 的 guide 执行具体转区操作
- **管理交互**: 处理检索、子问题拆解、LLM 调用

### 3. Level 2 的两种模式
| 模式 | 延迟 | Token | 用途 |
|:-----|:-----|:------|:-----|
| 轻量调用 | ~50ms | ~100 | 生成转区策略 (A/B/D区) |
| 完整调用 | ~100ms | ~200 | CI 仲裁验证 |

## 子问题处理与聚合机制

### 子问题生命周期完整流程

```
原始查询 (Zone A C1I1)
    ↓
协调器拆解 → 子问题列表 [SP1, SP2, SP3] (预期 C0I1)
    ↓
┌─────────────────────────────────────────────────────────────┐
│ 子问题 CI 判定流程                                           │
│                                                             │
│ 每个子问题:                                                  │
│ 1. 简化验证 (推荐): 轻量 Level 0 验证 C=0                     │
│    - 通过: 直接进入 Zone C 处理                              │
│    - 失败(C=1): 返回协调器重新拆解                           │
│                                                             │
│ 2. 或完全信任: 直接使用拆解时的预期 CI (C0I1)                 │
└─────────────────────────────────────────────────────────────┘
    ↓
子问题进入 Zone C 处理逻辑 → 产生子结果
    ↓
协调器收集所有子问题结果
    ↓
协调器决策输出策略:
  ├── 直接输出: 简单列表类回答，无需聚合
  ├── 等待聚合: 收集全部子问题后统一输出
  └── 二次整合: 需要综合推理，组装后重新走 Level 012 判定
    ↓
最终输出
```

### 输出策略对比

| 策略 | 适用场景 | 优点 | 缺点 |
|:-----|:---------|:-----|:-----|
| **直接输出** | 独立事实性问题 | 延迟低，用户体验好 | 缺乏整体连贯性 |
| **等待聚合** | 结构化列表回答 | 结果完整，格式统一 | 用户等待时间长 |
| **二次整合** | 需要综合分析的问题 | 回答质量高，逻辑连贯 | 额外 LLM 调用成本 |

**示例对比**:

```
查询: "设计一个电商系统需要考虑哪些方面？"

策略1-直接输出:
  - 子问题1回答: 数据库设计要点...
  [用户看到第一条]
  - 子问题2回答: 缓存策略...
  [用户看到第二条]
  [体验割裂]

策略2-等待聚合:
  [等待全部完成]
  1. 数据库设计: ...
  2. 缓存策略: ...
  3. 负载均衡: ...
  [完整列表，体验好]

策略3-二次整合:
  [收集所有子问题结果]
  [组装成新查询: "基于以下要点，给出电商系统设计方案..."]
  [重新判定 CI → Zone C 输出]
  [得到连贯的设计方案文档]
```

### 聚合责任归属：协调器 vs Zone C

**推荐方案：协调器负责聚合**

#### 为什么协调器负责更好？

| 维度 | 协调器负责 | Zone C 负责 |
|:-----|:-----------|:------------|
| **职责边界** | ✅ 协调器掌握全局上下文 | ❌ Zone C 只应处理单一问题 |
| **策略灵活性** | ✅ 可根据问题类型选择聚合策略 | ❌ Zone C 逻辑固定，难以扩展 |
| **递归风险** | ✅ 避免循环依赖 | ⚠️ Zone C 聚合可能产生递归复杂性 |
| **状态管理** | ✅ 协调器管理子问题状态 | ❌ Zone C 应无状态 |
| **调试维护** | ✅ 单一控制点，易于追踪 | ❌ 分散逻辑，难以调试 |

#### 关键设计原则

**Zone C 的定位**:
- 单一问题的直接回答单元
- 无上下文感知能力
- 不处理复杂的依赖关系

**协调器的定位**:
- 子问题的生命周期管理
- 全局状态维护
- 聚合策略决策

#### 完整数据流

```
协调器拆解子问题:
  SP1: "数据库设计要点" (预期 C0I1)
  SP2: "缓存策略" (预期 C0I1)
  SP3: "负载均衡" (预期 C0I1)
    ↓
对每个子问题:
  - 协调器提交到 Level 012 (简化验证)
  - 确认 C0I1 后，调用 Zone C 处理逻辑
  - Zone C 返回结果给协调器 (非直接输出)
    ↓
协调器收集: [Result1, Result2, Result3]
    ↓
协调器判断聚合方式:
  Case 1: 简单列表 → 格式化拼接 → 直接输出
  Case 2: 综合分析 → 组装新查询 → 重新走 Level 012 → Zone C 输出
```

### 与 Level 012 的交互细节

#### 子问题是否需要完整 Level 012 流程？

**方案对比**:

| 方案 | 延迟 | 适用场景 | 实现复杂度 |
|:-----|:-----|:---------|:-----------|
| **完整流程** | ~50-150ms | 高可靠性要求 | 低 |
| **简化验证** | ~1ms | 信任拆解质量 | 中 |
| **完全信任** | 0ms | 完全信任 LLM 拆解 | 低 |

**推荐：简化验证**
- 使用 Level 0 轻量验证 C=0
- 不触发 Level 1/2（节省成本）
- 验证失败返回协调器重新拆解

```python
# 简化验证伪代码
def validate_subproblem(sub_problem, expected_ci):
    """子问题轻量验证"""
    # 仅用 Level 0 验证复杂度
    l0_result = level0_router.quick_classify(sub_problem.query)
    
    if l0_result['C'] != expected_ci['C']:
        # 复杂度不符，返回协调器重新处理
        return {'valid': False, 'reason': 'C mismatch', 'actual': l0_result}
    
    return {'valid': True, 'ci': l0_result}
```
